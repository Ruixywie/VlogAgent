## 一、整体设计思路

### 1.1 核心范式

**Director-Editor-Critic 三 Agent 协作**——三个专业化 Agent 通过双层循环协作完成视频美学增强。

**关键设计原则**：VlogAgent 不是一个"固定流水线 + 参数搜索器"，而是一个**真正的 Agent 系统**——Director 自主制定风格策略和分阶段规划，Editor 为每阶段生成候选方案并通过视觉预筛选选出最优，Critic 结构化评审并做出路由决策（accept/refine/skip）。Agent 决定的不是"参数取多少"，而是"做什么 + 用什么工具 + 怎么做 + 做得好不好"。

视频相比图片多出时间维度和音频维度，核心应对策略：**将视频问题转化为元数据/文本问题**，Agent 在符号空间完成规划与评估，只在执行阶段才接触实际视频流。

### 1.2 与 PhotoAgent 的算法区别

| | PhotoAgent | VlogAgent |
|--|--|--|
| Agent 数量 | 1 个（全能 Agent） | 3 个（Director / Editor / Critic） |
| 循环结构 | 单层（Perceive→Plan→Execute→score→loop） | 双层（外层策略 + 内层逐阶段打磨） |
| 闭环信号 | 数字分数（score > threshold） | 结构化反馈（逐段因果分析 + 改进建议 + 路由决策） |
| 搜索策略 | 无约束 MCTS | Stage-Aware Visual Pre-screening（PIL 模拟 + CLIP+MLP） |
| 规划方式 | 无阶段概念，自由组合 | 5 阶段专业工作流（stabilize→denoise→color_correct→color_grade→sharpen） |
| 失败处理 | 回退到上一版本 | Critic 因果分析 → 针对性调整（refine）或跳过（skip） |
| 训练策略 | 训练 UGC Reward Model | 只训练 CLIP+MLP 快速评分（Director/Editor/Critic 用通用 VLM） |

### 1.3 为什么要分阶段规划？MCTS 自由搜索为什么不行？

#### PhotoAgent 的 MCTS 为什么在图片上可行

PhotoAgent 的 MCTS 自由搜索能工作，依赖三个条件：

1. **搜索空间小**：图片是单一目标，搜索空间 = N 个工具 × 参数范围。没有"片段"维度，不需要考虑段间一致性。
2. **VLM 评估有区分度**：PhotoAgent 每次 MCTS 模拟都调用 VLM API（3-5s/次）看实际画面评分。VLM 能感知视觉差异（"这张更通透" vs "这张过曝了"），所以不同方案的分数确实不同，UCB 能有效引导搜索方向。
3. **50 次模拟够用**：因为搜索空间小，50 次模拟足以覆盖有意义的子空间。

#### 为什么直接搬到视频上就失效了

把 MCTS 从图片迁移到视频，三个条件全部被打破：

**1. 搜索空间爆炸——多了"片段"维度**

图片 MCTS 的搜索空间：`{N 个工具} × {参数范围}`，大约几十种组合。

视频 MCTS 的搜索空间：`{N 个工具} × {M 个片段} × {参数范围} × {排列顺序}`。以 5 种工具、4 个片段为例，搜索空间比图片大了**数十倍以上**。50 次模拟根本不够覆盖。

**2. VLM 评估太贵——不得不退化为硬编码规则**

PhotoAgent 的 MCTS 之所以有效，核心在于每次模拟都用 VLM 做视觉评估。但视频有多个片段，每次模拟需要对多段分别评估。如果也用 VLM，50 次模拟 × M 段 = 数百次 VLM 调用，耗时和成本都不可接受。

因此我们的实现退而求其次，用**硬编码 if-else 规则**（如"亮度在合理范围内 → +0.1"）替代 VLM 评估。但规则评分无法感知视觉差异，结果是所有方案得分几乎相同（如 0.700 / 0.700 / 0.700），UCB 退化为随机选择，搜索形同虚设。

**这是根本矛盾**：用 VLM 评估 → 太贵太慢；用规则评估 → 没有区分度。PhotoAgent 能承受 VLM 评估的成本（图片单目标，50 次模拟 × 1 张图），但视频不行。

**3. 操作顺序依赖被搜索空间放大**

图片和视频编辑都存在操作顺序依赖（先锐化再降噪对两者都有害）。但图片 MCTS 因为搜索空间小、VLM 评估有区分度，即使探索了错误顺序的方案，VLM 也能通过低分将其淘汰。

视频 MCTS 由于评分无区分度，**无法区分顺序正确和顺序错误的方案**——两者得分一样。这导致错误顺序的方案不会被淘汰，进一步稀释了有效搜索。

#### 总结：核心矛盾

```
PhotoAgent MCTS（图片）：
  搜索空间 = 小（单目标 × N 工具）
  模拟评估 = VLM API（3-5s/次，但有区分度 ✓）
  50 次模拟 = 够用 ✓
  → 搜索有效

VlogAgent MCTS（视频，直接迁移）：
  搜索空间 = 大（M 段 × N 工具 × 排列）
  模拟评估 = 硬编码规则（快但无区分度 ✗）  ← 因为 VLM 评估太贵
  50 次模拟 = 不够用 ✗
  → 搜索退化为随机
```

**核心矛盾是"评估质量 vs 评估成本"**：视频的多片段结构使得高质量评估（VLM）成本过高，而低成本评估（规则）又没有区分度，MCTS 搜索无论怎么调参都无法有效工作。

#### 解决：Stage-Aware Planning（分阶段规划）

核心思路：**不在巨大空间中搜索，而是用专业知识将大问题分解为多个小问题，逐个解决**。

```
MCTS 自由搜索（旧）：
  搜索空间 = {所有工具} × {所有段} × {参数范围} × {排列顺序}  → 巨大
  评估方式 = 硬编码规则（无区分度）
  模拟次数 = 50 次（不够用）
  结果：所有方案得分相同，搜索无效

Stage-Aware Planning（新）：
  搜索空间 = 每阶段仅 2-3 条候选（Director 已决定做/不做）  → 极小
  评估方式 = PIL 视觉模拟 + CLIP+MLP 感知评分（有区分度）   → 解决评估矛盾
  决策方式 = 每阶段独立决策 + Critic 评审
  结果：每个阶段的候选分数有显著差异，选择有意义
```

具体来说：

1. **分阶段消除了搜索空间爆炸**：Director 看完整视频后，按专业工作流的 5 个阶段逐一决策（skip / global / per_segment + 方向描述）。每阶段只暴露该阶段的 1-2 种工具，Editor 只生成 2-3 条候选（参数不同），搜索空间从数百降到个位数。操作顺序由专业知识保证（stabilize → denoise → color_correct → color_grade → sharpen），不再需要搜索排列。

2. **PIL + CLIP+MLP 解决了"评估质量 vs 成本"的矛盾**：在关键帧上用 PIL 快速模拟每个候选的视觉效果（~5ms/帧），然后用训练过的 CLIP+MLP 模型打分（~10ms/帧）。速度接近硬编码规则，但区分度接近 VLM——因为 CLIP 编码了视觉语义，MLP 学习了美学偏好。这是 MCTS 不可能做到的：MCTS 需要在搜索过程中反复评估，评估成本被模拟次数放大；而 Stage-Aware Planning 每阶段只评估 2-3 个候选，总评估次数很少。

3. **阶段间解耦，失败可控**：某阶段 Critic 判定 degraded → 只跳过该阶段，不影响其他阶段已 accept 的操作。不像 MCTS 中一个坏动作可能污染整个方案。

4. **Critic 闭环提供 VLM 级别的评审**：PIL+CLIP+MLP 只用于快速预筛选（选出最有希望的候选），最终的质量把关由 Critic（VLM 看对比 Storyboard）完成。每阶段只需 1 次 VLM 评审调用（而非 MCTS 的 50 次），成本可控，但评估质量不打折。

### 1.4 形式化

```
VlogAgent 反思闭环：
  G = Director(s₀)                               ← 全局风格策略 + 分阶段规划
  while Director.should_continue():               ← 外层循环
    for stage in G.stages:                        ← 逐阶段处理
      if stage.scope == "skip": continue
      for attempt in range(max_refine + 1):       ← 阶段内 refine 循环
        candidates = Editor(sₜ, G, stage, Fₜ₋₁)  ← 生成 2-3 条候选
        best = PILSimulate + CLIP+MLP(candidates) ← 视觉预筛选
        sₜ₊₁ = FFmpeg(sₜ, best)                  ← 执行
        Fₜ = Critic(s₀, sₜ₊₁, best)             ← 评审 + 路由
        if Fₜ.route == "accept": break            ← 下一阶段
        elif Fₜ.route == "skip": break            ← 跳过此阶段
        else: continue                             ← refine，重新生成候选
    action_chain.extend(accepted_actions)          ← 累积动作
  final_execute(original_video, action_chain)      ← 原视频一次性编码
```


### 2.2 三个 Agent 的详细设计

#### 2.2.1 Director Agent（导演）— 通用 VLM，不训练

**职责**：
1. 初始化：观察视频全貌 → 制定 StyleBrief（含分阶段规划）
2. 外层循环：`should_continue()` 纯规则判断是否继续
3. redirect 时：根据 Critic 反馈调整风格策略（调 VLM）

**为什么不需要训练**：Director 的任务是创意性的（判断视频风格、情绪、方向），通用 LLM 擅长。调用频率低（一次运行 1-3 次），不是瓶颈。

**输出**：StyleBrief

```python
@dataclass
class StageDecision:
    stage: str       # stabilize/denoise/color_correct/color_grade/sharpen
    scope: str       # "skip" / "global" / "per_segment"
    direction: str   # 方向描述，如 "轻度降噪，保留纹理细节"
    target_segments: list[str]  # scope=per_segment 时指定段

@dataclass
class StyleBrief:
    overall_style: str      # "清新自然" / "电影感"
    color_direction: str    # "保持暖色调" / "偏冷"
    priority: str           # "优先修复暗段曝光"
    constraints: list[str]  # "不要对高速运动段做 stabilize"
    target_mood: str        # "宁静治愈"
    stages: list[StageDecision]  # 分阶段规划（5 个阶段）
```

**Stage-Aware Planning 的价值**：
- 顺序来自 Director 的**专业判断**（不是代码硬编码）
- 每阶段 Agent 自主决策：跳过 / 全局 / 段级
- color_correct（技术校正）和 color_grade（创意调色）分离，对齐专业工作流

**should_continue() 终止条件**（满足任一即终止）：
1. 达到最大外层迭代次数（默认 2）
2. Critic 的 overall_score > 0.85
3. Critic 没有 degraded 段且无 global_issues
4. 连续 2 轮外层循环无提升

#### 2.2.2 Editor Agent（编辑师）— 通用 VLM + 视觉预筛选，不训练

**职责**：在 Director 的分阶段规划下，为每个阶段生成候选方案并通过 PIL 模拟 + CLIP+MLP 预筛选最优。

**内部流程**：

```
对每个非 skip 阶段：
  ① Editor.suggest_for_stage()：
     VLM 看 Storyboard + Director 的阶段约束 + 阶段内 Critic 反馈
     → 生成 2-3 条候选 EditAction（只包含该阶段的工具）
     max_tokens=600

  ② StageSelector.select_best()：
     遍历候选 → PIL 模拟关键帧效果 → CLIP 提取特征 → MLP 打分
     → 返回最高分候选
     不可模拟的工具（stabilize 等）给 0.5 基准分
```

**实现文件**：
- `src/perceiver/perceiver.py` — Editor 的候选生成器（observe + suggest_for_stage）
- `src/planner/stage_selector.py` — 逐阶段视觉预筛选
- `src/planner/pil_simulator.py` — PIL 模拟模块

**PIL 模拟支持的工具**：

| 工具 | PIL 模拟方式 |
|------|-------------|
| color_correct/color_grade | ImageEnhance.Brightness/Contrast/Color + numpy gamma |
| white_balance | R/B 通道缩放 |
| denoise | GaussianBlur 近似 |
| sharpen | ImageEnhance.Sharpness |
| stabilize / speed_adjust | 不可模拟（返回 None，给 0.5 基准分） |

#### 2.2.3 Critic Agent（评审）— 通用 VLM + 技术指标

**职责**：
1. 对比编辑前后画面 → 逐段因果分析
2. 输出结构化反馈 + 改进建议
3. 路由决策 → accept / refine / skip

**两种评审模式**：

| 模式 | 方法 | 用途 |
|------|------|------|
| `evaluate()` | 完整评审（技术指标 + VLM 对比 Storyboard） | 外层循环的整体评审 |
| `evaluate_stage()` | 轻量评审（技术指标 + VLM 看对比图） | 逐阶段评审（当前主要使用） |

**输出**：CriticFeedback

```python
@dataclass
class SegmentCritic:
    segment_id: str
    verdict: str      # "improved" / "unchanged" / "degraded"
    reason: str       # 因果分析

@dataclass
class CriticFeedback:
    overall_score: float
    segment_feedback: dict[str, SegmentCritic]  # 逐段反馈
    global_issues: list[str]
    global_positives: list[str]
    suggestions: list[str]        # 改进建议
    route: str                    # "accept" / "refine" / "skip"
    route_reason: str
```

**阶段级路由决策逻辑**（VLM 自主决定，规则兜底）：

```
VLM 看对比 Storyboard → 逐段评审 → 自主决定路由：
  - accept: 大多数段 improved 且没有 degraded
  - refine: 有 degraded 段，需要调整
  - skip:   所有段 unchanged（无效果）或操作有害

VLM 未给有效路由时的规则兜底：
  - refine_count >= 2 → skip（重试过多，跳过）
  - 其他 → refine
```

**Critic 综合评分权重**（阶段级评审）：
- 视觉质量：15%
- 内容保真：20%
- 美学（VLM）：65%

### 2.3 核心设计决策

#### 2.3.1 Storyboard 视觉压缩

将密集关键帧（30-40 张）拼成 **1 张 Storyboard 网格图**，token 消耗从 ~100K 降到 ~3K（减少 15 倍），解决了 72B 模型输入过大的问题。

Critic 评审使用**对比 Storyboard**：左列原始帧，右列编辑后帧，一张图完成所有段的对比。

#### 2.3.2 搜索与最终执行分离 + 段级编辑

- **搜索阶段**：在当前视频上执行（CRF=18），结果仅用于评估
- **最终执行**：`_sort_by_stage()` 按阶段排序 → 滤镜合并 → CRF=10 单次编码
- **段级编辑**：`enable='between(t,start,end)'` 仅对目标段生效
- **stabilize 限制**：仅 global 模式，段级自动跳过（`global_only=True`）

#### 2.3.3 Evaluator 权重（美学为主）

| 维度 | 方法 | 权重 |
|------|------|------|
| 整体美学 | VLM 对比 Storyboard 打分 | **60%** |
| 内容保真 | CLIP 余弦相似度 | 15% |
| 视觉质量 | Laplacian + 曝光检查 | 10% |
| 段间一致 | Lab 色彩距离 | 10% |
| 音频完整 | 时长对比 | 5% |

#### 2.3.4 CLIP+MLP 快速评分模型

用于 StageSelector 的视觉预筛选（替代 MCTS 中的 VLM 评估）。

```
状态：已训练，验证准确率 81.1%
用途：逐阶段候选预筛选（PIL 模拟后打分）
速度：<10ms/次
部署：StageSelector.select_best()
模型：CLIP ViT-B-32 + 2 层 MLP
```

### 2.4 VLM 调用统计（典型运行）

以外层 2 轮、每阶段 1-2 次为例：

| 阶段 | 每轮调用 | 说明 |
|------|---------|------|
| Director 策略 | 1 次 | 制定 StyleBrief + 分阶段规划 |
| Editor 观察 | 1 次 | 观察 Storyboard（外层复用） |
| Editor 逐阶段候选 | 3-5 次 | 每个非 skip 阶段 1 次 |
| Critic 逐阶段评审 | 3-5 次 | 每个非 skip 阶段 1 次 |
| PIL+CLIP+MLP 预筛选 | 0（本地） | 毫秒级，无 VLM 调用 |
| **合计** | **~8-12 次** | 远少于旧架构的 ~60 次 |

### 2.6 已解决的问题

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| MCTS 搜索无效（所有方案分数相同） | 硬编码 if-else 评分无区分度 | 改为逐阶段 PIL 模拟 + CLIP+MLP 视觉预筛选 |
| stabilize 导致锯齿 | 启发式权重过高被反复选中 | 默认 skip + global_only 限制 |
| VLM 返回中文工具名 | "视频降噪"、"smoothing" 等 | TOOL_NAME_MAP 中英文映射 |
| VLM 参数名错误 | smoothing_value vs smoothing | PARAM_NAME_FIX 参数名纠正 |
| 无效操作被 Critic accept | 工具名不在 registry | 执行成功检查 + 工具名验证 |
| Critic 反馈跨阶段污染 | 阶段间共享 feedback 变量 | 每阶段独立 stage_critic_feedback |
| 116K token 输入崩溃 | 逐帧发送太多图片 | Storyboard 网格图压缩（15 倍） |
| 多次编码画质损失 | 每步工具独立编码 | 搜索与最终执行分离 + 滤镜链合并 |
| Evaluator 过于死板 | 机械指标占 80% | 美学占 60%（对齐 PhotoAgent） |
| NPU vLLM 启动失败 | ACL Graph 编译 HCCL 初始化失败 | 安装 vllm_ascend 0.17.0rc1（已修复图编译问题） |
