# VlogAgent 实施计划

> 以论文发表为导向，从单视频美化（方向 A）做起，逐步扩展到多素材剪辑（方向 B）。

---

## 一、整体设计思路

### 1.1 核心范式

**完全对齐 PhotoAgent 的 Agent 化设计**——Perceiver → Planner → Executor → Evaluator 的闭环架构，迁移到视频领域。

**关键设计原则**：VlogAgent 不是一个"固定流水线 + 参数搜索器"，而是一个**真正的 Agent**——它自主感知视频问题、自主决定编辑策略、自主选择工具执行、自主评估结果。Agent 决定的不是"参数取多少"，而是"做什么 + 用什么工具 + 怎么做"。

同时，视频相比图片多出时间维度和音频维度，需要解决维度爆炸、动态美学量化、闭环成本三大问题。核心应对策略：**将视频问题转化为元数据/文本问题**，Agent 在符号空间完成规划与评估，只在执行阶段才接触实际视频流。

### 1.2 两阶段推进策略

```
阶段 A：单视频美化（基础模块）
  输入：一段原始视频（可能包含多镜头、多光照、多场景）
  输出：美化后的同一段视频
  价值：验证 MCTS + 闭环在视频上的可行性，积累视频美学评估经验
        ↓
        作为子模块嵌入
        ↓
阶段 B：多素材剪辑拼接（论文主体）
  输入：多段原始视频素材 + 简单主题描述
  输出：筛选、排序、拼接、美化后的完整 Vlog
  价值：核心创新（元数据空间 MCTS + 分解式奖励 + 分级代理评估）
```

### 1.3 论文定位

- **目标会议**：ACM Multimedia / AAAI / CVPR
- **暂定标题**：VlogAgent: Agentic Vlog Editing via Metadata-Space Planning and Tiered Proxy Evaluation
- **核心贡献**：
  1. 问题定义：首次定义"全自动多素材 Vlog 剪辑"任务
  2. 元数据空间 MCTS 规划
  3. 分解式奖励模型（5 维子指标）

## 二、阶段 A：单视频美化模块（详细设计）

### 2.1 问题定义

用户提供**一段原始视频**（如手机拍摄的 1-5 分钟片段），这段视频可能包含：
- 多个镜头/场景（室内→室外，白天→傍晚）
- 不同区间的亮度/色温/曝光差异
- 局部抖动、失焦、噪点
- 无处理的原始音频

VlogAgent-A 需要自动对其进行美学增强，输出一段视觉统一、质量提升的视频。

### 2.2 Agent 化系统架构（对齐 PhotoAgent）

#### 2.2.1 PhotoAgent 的四组件架构 → VlogAgent 的对应迁移

```
PhotoAgent（图片）                    VlogAgent（视频）
─────────────                        ─────────────
Perceiver（VLM 观察图片             Perceiver（VLM 观察视频关键帧
  → 生成 N 条候选编辑指令）            + 分段元数据 → 生成 N 条候选编辑指令）

Planner（MCTS 搜索最优               Planner（MCTS 搜索最优
  动作序列）                           动作序列，在元数据空间进行）

Executor（按指令类型                  Executor（按指令类型
  路由到 OpenCV/PIL 或 Flux）          路由到 FFmpeg 或 AI 视频工具）

Evaluator（CLIP + LAION              Evaluator（DOVER + CLIP
  + ImageReward + UGC 评分）           + MLLM-as-Judge + 一致性检查）
```

#### 2.2.2 系统流水线

```
输入：一段原始视频
       │
       ▼
┌──────────────────────┐
│  Perceiver: 视频感知    │  场景分割 + 质量检测 + VLM 观察

└──────────┬───────────┘
           │  候选指令集（语义级，如"提亮室内段""稳定抖动镜头""天空换蓝天"）
           ▼
┌──────────────────────┐
│  Planner: MCTS 规划    │  搜索最优动作序列
│                        │  （做什么 → 用什么工具 → 先后顺序）
└──────────┬───────────┘
           │  Top-K 动作序列
           ▼
┌──────────────────────┐
│  Executor: 工具执行     │  Tool Selection → 路由到对应工具
│                        │  FFmpeg / AI 视频模型 / OpenCV

           ▼
```

#### 2.2.3 异构工具池（Tool Pool）

类比 PhotoAgent 的 "Basic Tools + AI Tools"，VlogAgent 的 Executor 拥有以下工具池：

```
VlogAgent 工具池
│
├── Basic Tools（传统视频处理，快速、确定性强）
│   ├── FFmpeg 滤镜
│   │   ├── color_adjust(brightness, contrast, saturation)  色彩调整
│   │   ├── white_balance(temperature)                       白平衡
│   │   ├── denoise(strength)                                降噪
│   │   ├── sharpen(amount)                                  锐化
│   │   ├── stabilize(smoothing)                             稳定化
│   │   ├── speed_adjust(factor)                             变速
│   │   ├── crop(x, y, w, h)                                裁剪/重构图
│   │   ├── lut_apply(lut_file)                              LUT/滤镜
│   │   └── transition(type, duration)                       转场效果
│   │
│   └── OpenCV / PIL
│       ├── rotate(angle)                                    旋转
│       ├── flip(direction)                                  翻转
│       └── text_overlay(text, position, style)              文字叠加
│
├── AI Tools（AI 生成/编辑模型，能力强但速度慢、成本高）
│   ├── 视频风格迁移（Video Style Transfer）
│   │   └── 将片段风格化为电影感/动漫风/复古等
│   ├── 视频修复（Video Inpainting / Object Removal）
│   │   └── 移除画面中的路人/杂物/水印
│   ├── 天空替换（Sky Replacement）
│   │   └── 将阴天天空替换为蓝天白云/晚霞
│   ├── 超分辨率（Video Super Resolution）
│   │   └── 将低分辨率片段提升到高清
│   ├── AI 调色（AI Color Grading）
│   │   └── 基于参考图片或文字描述的智能调色
│   ├── 背景虚化（AI Bokeh / Depth-based Blur）
│   │   └── 模拟浅景深效果，突出主体
│   └── AI 音频增强（Audio Enhancement）
│       ├── 语音降噪 / 风声去除
│       └── BGM 智能匹配与节拍对齐
│
└── Compound Tools（组合工具，封装常见多步操作）
    ├── auto_color_harmonize(anchor_seg)    全局色彩统一
    ├── smooth_transition(seg_a, seg_b)     段间过渡平滑
    └── segment_enhance(seg_id, style)      单段综合增强
```

#### 2.2.4 Tool Selection：动态路由机制

Perceiver 生成的每条编辑指令是**语义级**的自然语言，Tool Selection Module 将其路由到合适的工具：

```
编辑指令（Perceiver 生成）                  → 路由到的工具
──────────────────────────────────────────────────────────
"提高第 2 段的亮度和对比度"                  → FFmpeg: color_adjust
"稳定第 1 段的抖动镜头"                     → FFmpeg: stabilize
"给全片应用电影感色调"                       → FFmpeg: lut_apply  或  AI: 风格迁移
"移除第 3 段背景中的路人"                    → AI: Video Inpainting
"将第 4 段的阴天天空替换为晚霞"              → AI: Sky Replacement
"对模糊的第 5 段进行清晰化"                  → AI: Super Resolution
"添加浅景深效果突出人物"                     → AI: Bokeh
"统一全片色彩风格"                          → Compound: auto_color_harmonize
"在第 2 段和第 3 段之间添加淡入淡出转场"     → FFmpeg: transition
```

路由逻辑：
```python
def select_tool(instruction: str, segment_metadata: dict) -> Tool:
    """LLM 驱动的工具选择"""
    prompt = f"""
    你是视频编辑工具选择专家。根据编辑指令选择最合适的工具。

    编辑指令：{instruction}
    片段信息：{segment_metadata}

    可用工具：
    - FFmpeg 系列：色彩调整、白平衡、降噪、锐化、稳定化、变速、裁剪、LUT、转场
    - AI 系列：风格迁移、物体移除、天空替换、超分辨率、AI调色、背景虚化、音频增强
    - 组合工具：全局色彩统一、段间过渡平滑、单段综合增强

    选择原则：
    1. 优先使用 Basic Tools（快速、确定性强）
    2. 仅在需要语义理解/内容生成时使用 AI Tools
    3. 返回工具名称和参数

    以 JSON 格式输出。
    """
    return llm.call(prompt)
```

#### 2.2.5 关键对比：旧设计 vs 新设计

```
旧设计（自动化工具）：                   新设计（真正的 Agent）：
  固定 8 步 Pipeline                      Perceiver 自主观察 → 生成开放式指令
  → 每步搜参数值                          → Planner MCTS 搜索动作序列
  → 只有 FFmpeg                           → Tool Selection 动态路由
  → 只能调参数                            → 异构工具池（FFmpeg + AI 模型）
                                          → 既能调参数，也能做语义编辑
  Agent 说："亮度设为 +20"                Agent 说："这段室内太暗了，需要提亮
                                            并修正偏黄的白平衡；另外背景
                                            有个路人影响构图，需要移除"
```

这使得 VlogAgent 的编辑能力从**纯参数调整**升级为**参数调整 + 语义编辑**，与 PhotoAgent 的能力对齐。

### 2.3 Step 1：视频分析（Video Analysis）

#### 2.3.1 场景分割

一段原始素材往往包含多个视觉风格不同的片段（如从室内走到室外）。必须先做场景分割，否则全局统一调色会顾此失彼。

```
原始视频：[====室内暖光====|==过渡走廊==|====室外日光====|==傍晚暮色==]
                ↓ PySceneDetect / TransNetV2
分割结果：[  Seg-1 (0-12s) | Seg-2 (12-15s) | Seg-3 (15-35s) | Seg-4 (35-50s) ]
```

**实现方案**：
- 主方案：PySceneDetect 的 ContentDetector（基于帧间差异的自适应阈值）
- 备选：TransNetV2（学习式镜头边界检测，精度更高但需要 GPU）
- 对于渐变式场景变化（如日落过程），需要设置较低的阈值或使用滑窗式特征分析

**可能的问题 & 解法**：

| 问题 | 表现 | 解法 |
|------|------|------|
| 过度分割 | 手持抖动导致帧间差异大，误判为镜头切换 | 先做视频稳定化预处理；或调高 ContentDetector 阈值 |
| 分割不足 | 缓慢的光照变化（如日出）未被检测到 | 增加亮度/色温趋势检测：滑窗计算色彩统计量，变化超过阈值则强制分割 |
| 过渡段处理 | 场景切换不是瞬时的（如平移镜头从室内移向窗外） | 将过渡段标记为"过渡区间"，不做独立调色，而是对前后段参数做线性插值 |

#### 2.3.2 逐段质量检测

对每个分割出的片段，提取以下指标：

```python
class SegmentMetadata:
    seg_id: int
    time_range: (float, float)        # 起止时间
    # ── 亮度与曝光 ──
    mean_brightness: float             # Y 通道均值 (0-255)
    brightness_std: float              # 帧间亮度波动（闪烁检测）
    overexposed_ratio: float           # 过曝像素占比
    underexposed_ratio: float          # 欠曝像素占比
    # ── 色彩 ──
    color_temp_est: float              # 估算色温 (K)
    mean_saturation: float             # 平均饱和度
    color_cast: (float, float)         # Lab 空间 a/b 偏移（偏色检测）
    # ── 清晰度与稳定性 ──
    sharpness_score: float             # Laplacian 方差（模糊检测）
    stability_score: float             # 光流场方差（抖动检测）
    noise_level: float                 # 噪点估计
    # ── 内容与音频 ──
    semantic_description: str          # MLLM 生成的场景描述
    has_speech: bool                   # 是否包含语音
    speech_text: str                   # ASR 转录文本
    audio_energy: float                # 音频能量（静音/正常/嘈杂）
    emotion_tag: str                   # 情感标签（欢快/宁静/紧张等）
```

**可能的问题 & 解法**：

| 问题 | 表现 | 解法 |
|------|------|------|
| 帧间亮度剧烈波动 | 经过闪光灯区域、警灯闪烁 | brightness_std 超过阈值时标记为"闪烁区间"，后续平滑处理而非简单调色 |
| 混合光源 | 同一段中左半画面暖光右半画面冷光 | 提取分区域色彩统计（上/下/左/右四象限），检测空间色彩不均匀度 |
| 运动模糊 vs 失焦模糊 | Laplacian 方差都低，但处理方式不同 | 运动模糊用稳定化缓解，失焦模糊无法修复→标记为"质量受限" |

#### 2.3.3 音频分析

```
音频分析管线：
  原始音频 → Whisper ASR → 语音区间 + 转录文本
           → librosa → 节拍检测 + 能量曲线 + 频谱特征
           → 分类器 → 音频类型标注（语音/音乐/环境音/静音）
```

**关键输出**：语音区间标记——后续编辑中有语音的区间不能做变速处理，也不应用 BGM 覆盖。

### 2.4 Step 2：编辑规划（Edit Planning）

#### 2.4.1 Agent 化的动作空间：从"参数搜索"到"语义指令"

**对齐 PhotoAgent 的设计**：Perceiver 观察视频后，自主生成一组**语义级**的候选编辑指令，而非在预定义参数表上搜索。

**Perceiver 的工作方式**：

```python
def perceive(video_path: str, segment_metadata: list[SegmentMetadata]) -> list[str]:
    """
    VLM 观察视频关键帧 + 分段元数据，自主生成候选编辑指令
    """
    # 提取每段的代表帧 → 拼成 storyboard
    storyboard = generate_storyboard(video_path, segment_metadata)

    prompt = f"""
    你是专业视频调色师和剪辑师。请观察这段视频的关键帧和分析数据，
    提出你认为最能提升视频美学质量的编辑建议。

    视频分析数据：
    {format_metadata(segment_metadata)}

    要求：
    1. 每条建议是一个独立的、原子性的编辑动作
    2. 指明作用于哪个片段（或全局）
    3. 既可以是参数调整类（色彩/曝光），也可以是语义编辑类（移除物体/替换天空）
    4. 按优先级排序（最重要的排最前）
    5. 生成 8-15 条候选指令

    以 JSON 数组格式输出，每条包含 action_description 和 target_segment。
    """
    return vlm.call(prompt, images=[storyboard])
```

```json
[
  {"action": "稳定第1段的抖动镜头",                "target": "seg-1", "type": "basic"},
  {"action": "提高第2段室内场景的亮度和白平衡",      "target": "seg-2", "type": "basic"},
  {"action": "移除第3段背景中的路人",               "target": "seg-3", "type": "ai"},

  {"action": "对第2段添加浅景深效果突出美食特写",     "target": "seg-2", "type": "ai"},
  {"action": "全片应用统一的暖色电影感色调",          "target": "global", "type": "basic"},
  {"action": "降低第1段的环境噪音",                 "target": "seg-1", "type": "ai"},
  {"action": "对第5段慢镜头处理增强氛围感",          "target": "seg-5", "type": "basic"},
  {"action": "统一全片色彩一致性",                   "target": "global", "type": "compound"},
```

**与旧设计的对比**：

```

  动作 = "brightness: +20"            动作 = "提高第2段室内场景的亮度和白平衡"
  → 机械地遍历参数空间               → Agent 理解"室内偏暗"的问题本质
```

#### 2.4.2 MCTS 规划：搜索最优动作序列

**MCTS 在 Agent 化设计中的定义**：

```
状态（State）：当前视频的编辑状态（已执行的动作列表 + 当前评分）
动作（Action）：从 Perceiver 生成的 N 条候选指令中选择一条执行
奖励（Reward）：Evaluator 的综合评分（多维美学指标）
终止条件：达到最大编辑步数（如 depth=5）或连续无提升
```

与 PhotoAgent 完全对齐：
- PhotoAgent：每一步从 N 条候选中选一条 → 执行 → 评估 → 搜索深度 3-5
- VlogAgent：每一步从 N 条候选中选一条 → 执行 → 评估 → 搜索深度 3-5

**树结构示意**：

```
                          根节点（原始视频, score=3.2）
                         /          |           \
                  动作1:稳定      动作2:提亮      动作3:移除路人
                  seg-1          seg-2           seg-3
                  score=3.5      score=3.8       score=3.6
                 /    \          /    \
           动作2:提亮  动作5:LUT  动作3:移除  动作6:天空替换
           score=4.0   score=3.9  score=4.1   score=4.3 ← 最优路径
                                               |
                                          动作9:统一色彩
                                          score=4.5
```

**一轮 MCTS 迭代的具体流程**：

```
═══════════════════════════════════════════════
迭代 #10
═══════════════════════════════════════════════

【Selection】 UCB1 选择路径
  根 → "提亮 seg-2"(avg=4.1, visits=5) ← 选中（高均值）
     → "天空替换 seg-4"(avg=4.3, visits=3) ← 选中
     → 到达未完全展开的节点

【Expansion】
  在 "天空替换" 节点下展开新动作 → "全片统一色彩"
  创建子节点

【Simulation（LLM 启发式 Rollout）】
  从 "统一色彩" 出发，LLM 推荐后续动作序列：
    → "锐化 seg-3" → "慢镜头 seg-5" → 终止

  快速评估（不渲染完整视频）：
    对每步动作，在关键帧上模拟效果：
    - Basic Tools 动作：numpy/PIL 直接在帧上应用参数 → DOVER 评分
    - AI Tools 动作：LLM 预估效果分数（不实际执行 AI 模型）
    → 综合 R = 4.5

【Backpropagation】
  沿路径回传 4.5
```

**Simulation 中的关键优化——对 AI 工具动作的快速评估**：

```
Basic Tools 动作（如色彩调整、降噪）：
  → 可以在关键帧上直接用 numpy 模拟 → 真实评分
  → 耗时 ~0.5s

AI Tools 动作（如天空替换、物体移除）：
  → 实际执行 AI 模型太慢（10-60s），不适合在 Simulation 中做
  → 替代方案：LLM 预估效果分数
    "如果把这段阴天天空替换为蓝天白云，基于当前画面构图，
     预估视觉质量提升多少分？"
  → 耗时 ~1-2s（一次 LLM 调用）
  → 有偏差，但 MCTS 多轮迭代可以容错
```

**搜索结束后 → Top-K 执行**：

```
MCTS 选出 Top-K（如 K=3）个最优动作序列：
  序列 1: 提亮seg-2 → 天空替换seg-4 → 统一色彩 → score=4.5
  序列 2: 稳定seg-1 → 提亮seg-2 → 移除路人seg-3 → score=4.3
  序列 3: 提亮seg-2 → LUT全片 → 锐化seg-3 → score=4.2

Executor 实际执行 Top-K 序列（此时才真正调用 AI 工具）
Evaluator 对 K 个实际结果打分 → 保留最高分版本
```

#### 2.4.3 操作顺序：从"固定 Pipeline"到"Agent 自主 + 软约束"

旧设计中的固定 Pipeline Order 仍然有价值（操作不可交换是物理事实），但在 Agent 化设计中改为**软约束**而非硬规则：

```
硬规则（不可违反）：
  - 稳定化必须在调色之前
  - 降噪必须在锐化之前
  - 曝光校正应在 LUT 之前

软约束（Perceiver/Planner 的知识）：
  - 通过 Prompt 让 Perceiver 了解操作依赖关系
  - Planner 在 MCTS 中对违反顺序的序列给予惩罚（降低奖励）
  - 但不完全禁止——某些场景下打破常规可能有创意效果
```

Perceiver 的 Prompt 中包含操作顺序知识：
```
注意以下编辑顺序原则（来自影视后期工业标准）：
- 稳定化/降噪等修复类操作应优先于调色类操作
- 曝光/白平衡校正应在 LUT/风格化之前
- 锐化应在降噪之后
- 语义编辑（物体移除/天空替换）建议在色彩调整之前，
  避免调色后再编辑导致色彩不一致
请在生成候选指令时考虑合理的执行顺序。
```

**可能的问题 & 解法**：

| 问题 | 表现 | 解法 |
|------|------|------|
| Perceiver 生成冗余/矛盾指令 | "提亮" 和 "降低曝光" 同时出现 | LLM 后处理去重去矛盾；或在 Planner 中自然淘汰矛盾路径 |
| AI 工具执行失败/效果差 | 天空替换生成了不自然的接缝 | Evaluator 打低分 → 自动回退；Executor 对 AI 工具结果做质量检查 |
| AI 工具 Simulation 评分不准 | LLM 预估 +0.5 分，实际 -0.2 分 | Top-K 策略缓冲：不只执行最优序列，而是执行 Top-3 实际对比 |
| 动作序列过长导致质量退化 | 5 步编辑后画面过度处理 | 最大步数限制（depth=3-5）+ 连续无提升早停 |
| 操作顺序不当 | 先加 LUT 再调曝光导致色彩异常 | 软约束惩罚 + Evaluator 检测并回退 |

#### 2.4.4 关键问题：单视频内部前后不一致的处理

这是单视频美化的**核心难题**。一段素材前半段偏暗后半段过亮，如果 Agent 只对某一段发出"提亮"指令，结果会导致各段风格不统一。

**在 Agent 化设计中如何自然解决**：

```
Perceiver 的感知能力使其能主动发现一致性问题：

第 1 轮迭代：
  Perceiver 观察全片关键帧 + 逐段元数据
  → 生成候选指令：
    "提亮 seg-1（室内段）"
    "降低 seg-3 曝光（过曝段）"
    "统一全片色彩一致性"        ← Perceiver 自主发现不一致问题
  → Planner MCTS 搜索 → 可能优先执行分段调色

第 2 轮迭代：
  Perceiver 观察编辑后的视频
  → 发现各段独立调色后风格不统一
  → 生成指令："统一全片色彩一致性"
  → Executor 调用 Compound Tool: auto_color_harmonize
```

**Compound Tool `auto_color_harmonize` 的三种内部策略**：

| 策略 | 实现 | 适用场景 |
|------|------|---------|
| **锚点对齐** | 选 DOVER 评分最高段为锚点 → Reinhard Color Transfer 对齐（α=0.5） | 各段差异中等，有一段质量明显最好 |
| **全局 LUT 统一** | 全视频套同一 LUT → 各段微调 | 需要统一风格感（旅行/电影感） |
| **段间过渡平滑** | 相邻段交界处前后各 1-2s 对色彩参数做线性渐变 | 配合上述策略使用，消除硬切跳变 |

**Agent 的策略选择**：Tool Selection Module 根据 Perceiver 指令的具体措辞和 Evaluator 的一致性评分，自动选择合适的策略。不需要硬编码，由 Agent 自主判断。

**Evaluator 的一致性检测**：Evaluator 包含专门的一致性评估维度，如果检测到段间亮度/色温跳变超过阈值，会自动在下一轮 Perceiver 中注入"一致性问题"的提示，引导 Agent 生成统一色彩的指令。

### 2.5 Executor：异构工具池 + 动态路由执行

#### 2.5.1 执行架构

对齐 PhotoAgent 的 Executor 设计：不绑定单一工具，而是维护一个**异构工具池**，根据编辑指令的类型动态路由到最合适的工具。

```
Planner 输出 Top-K 动作序列
     │
     ▼
对序列中的每个动作：
  1. Tool Selection Module 判断动作类型
     ├── Basic（色彩/稳定/变速/裁剪...）→ FFmpeg / OpenCV
     ├── AI（物体移除/天空替换/风格迁移...）→ AI 视频模型
     └── Compound（全局色彩统一...）→ 组合工具
  2. Executor 调用对应工具执行该动作
  3. 得到编辑后的视频
  4. 进入下一个动作（对上一步的输出继续编辑）
     │
     ▼
K 个序列各产生一个最终版本
Evaluator 对 K 个版本打分 → 保留最高分
```

#### 2.5.2 Basic Tools：FFmpeg / OpenCV 封装

对于参数调整类动作，Tool Selection 将语义指令翻译为 FFmpeg 命令：

```
Agent 指令                          → FFmpeg 命令
───────────────────────────────────────────────
"提高第2段亮度"                      → eq=brightness=0.08
"校正第1段白平衡，减少黄色偏色"       → colortemperature=temperature=6200
"稳定第3段的抖动镜头"                → vidstabdetect + vidstabtransform
"全片应用电影感 LUT"                 → lut3d=file=cinematic.cube
"第5段做 1.5 倍慢镜头"              → setpts=1.5*PTS + atempo=0.67
```

FFmpeg 滤镜速查：

| 操作 | FFmpeg 实现 | 备注 |
|------|------------|------|
| 曝光 | `eq=brightness=0.08` | 范围 -1.0 到 1.0 |
| 色温 | `colortemperature=temperature=5500` | 单位 K |
| 对比度/饱和度 | `eq=contrast=1.2:saturation=1.3` | 1.0 为原始 |
| 降噪 | `hqdn3d=4:4:4:4` | 时空域联合降噪 |
| 稳定化 | `vidstabdetect` → `vidstabtransform` | 两步操作 |
| LUT | `lut3d=file=xxx.cube` | 支持 .cube / .3dl |
| 变速 | `setpts=0.5*PTS` + `atempo=2.0` | 音频需同步处理 |
| 锐化 | `unsharp=5:5:1.0:5:5:0` | 在降噪之后 |
| 裁剪 | `-ss 00:00:12 -to 00:00:35` | 精确到帧级 |
| Crossfade | `xfade=transition=fade:duration=0.5` | 多种转场类型 |

**关键设计点**：
- 同一段上的多个 Basic Tools 动作可以合并为**单次 FFmpeg 调用**（`-vf "f1,f2,f3,..."`），避免多次编解码的质量损失
- 音频轨默认直接复制（`-c:a copy`），变速时单独处理

#### 2.5.3 AI Tools：视频级 AI 模型

对于语义编辑类动作，路由到 AI 视频处理模型：

| AI 工具 | 功能 | 候选模型/方案 | 耗时估计 |
|---------|------|-------------|---------|
| 视频物体移除 | 移除路人/杂物/水印 | ProPainter, E2FGVI | 10-30s/段 |
| 天空替换 | 替换天空为蓝天/晚霞 | SkyAR + 扩散模型 | 15-40s/段 |
| 视频风格迁移 | 电影感/动漫/复古 | ReReVST, CoCoNet | 20-60s/段 |
| 超分辨率 | 低分辨率→高清 | Real-ESRGAN Video, BasicVSR++ | 30-120s/段 |
| 背景虚化 | 模拟浅景深 | 深度估计(MiDaS) + 分层模糊 | 5-15s/段 |
| AI 调色 | 参考图片/文字风格调色 | 3D LUT 学习 / CLIP 引导调色 | 5-10s/段 |
| 语音降噪 | 去除风声/环境噪音 | DeepFilterNet, RNNoise | 2-5s/段 |

**AI 工具的调用原则**：
1. **优先 Basic Tools**：能用 FFmpeg 解决的不调 AI 模型（快 10-100 倍）
2. **AI 工具仅用于语义编辑**：移除物体、替换内容、生成新元素等 FFmpeg 做不到的操作
3. **失败处理**：AI 工具可能生成伪影/不自然结果 → Evaluator 检测到质量下降 → 自动回退

#### 2.5.4 Compound Tools：组合工具

封装常见的多步操作为单一可调用工具：

```python
def auto_color_harmonize(video_segments: list, anchor_id: int = None) -> list:
    """全局色彩统一：选锚点 → Reinhard Color Transfer → 段间过渡平滑"""
    if anchor_id is None:
        anchor_id = select_best_segment(video_segments)  # DOVER 评分最高
    anchor_stats = extract_lab_stats(video_segments[anchor_id])
    for seg in video_segments:
        if seg.id != anchor_id:
            apply_reinhard_transfer(seg, anchor_stats, strength=0.6)
    smooth_transitions(video_segments)
    return video_segments
```

#### 2.5.5 分段处理 + 拼接流程

```
对每个编辑动作（按 Planner 输出的序列逐步执行）：
  1. Tool Selection 判断工具类型
  2. 如果是 Basic Tool → FFmpeg 执行（尽量链式合并）
  3. 如果是 AI Tool → 调用对应 AI 模型
  4. 如果是 Compound Tool → 执行组合逻辑

所有动作执行完毕后：
  5. 段间过渡处理（自适应转场策略）
  6. FFmpeg concat 无缝拼接
  7. 音频连续性确保
```

**可能的问题 & 解法**：

| 问题 | 表现 | 解法 |
|------|------|------|
| 拼接处有帧重复或丢帧 | 视频在切点处卡顿 | 使用精确帧级切割（-copyts），避免关键帧对齐偏差 |
| 变速后音画不同步 | 语音加速但口型对不上 | 有语音区间禁止变速；或使用 rubberband 进行变速不变调处理 |
| LUT 在不同曝光段效果差异大 | 暗部应用 LUT 后过暗，亮部正常 | 在应用 LUT 前先做曝光标准化（将所有段归一到类似亮度范围），应用 LUT 后再微调 |
| 稳定化导致画面裁切 | 边缘出现黑边或画面缩小 | 使用 vidstabtransform 的 zoom 参数自动缩放填充 |
| 多次编码质量损失 | 反复解码-编码导致画质下降 | 尽量用单次 FFmpeg 命令链式处理（-filter_complex），避免多次落盘 |

### 2.6 Evaluator：质量评估与闭环驱动

#### 2.6.1 评估维度

对齐 PhotoAgent 的 Evaluator（CLIP + LAION + ImageReward + UGC Score），VlogAgent 的 Evaluator 采用**多维集成评估**：

```
VlogAgent Evaluator 评估维度：

  ① 视觉质量（与原始对比）
     - DOVER/Q-Align 逐帧评分的提升幅度
     - 过曝/欠曝比例的变化
     - 清晰度（Laplacian 方差）变化

  ② 内容保真度
     - CLIP 相似度：编辑前后的语义一致性（避免 AI 工具过度修改内容）
     - 人脸/关键物体的保持度检查

  ③ 段间一致性
     - 相邻段亮度差异：|mean_brightness[i] - mean_brightness[i+1]|
     - 相邻段色温差异、饱和度差异
     - 交界帧处的 SSIM（结构相似性）
     → 综合为 consistency_score

  ④ 音频完整性
     - 语音区间是否被破坏（变速/裁切）
     - 音量标准化是否合理

  ⑤ 整体美学评分
     - MLLM-as-Judge：对处理前后的关键帧拼图做 A/B 对比评分
```

#### 2.6.2 闭环驱动机制（对齐 PhotoAgent）

对齐 PhotoAgent 的闭环逻辑：Evaluator 的评分决定是否继续编辑、是否回退。

```
每轮 Executor 执行完毕后：

  Evaluator 对 Top-K 个结果打分：
    best_score = max(scores of K candidates)
    prev_score = 上一轮最佳分数

    if best_score > prev_score:
      # 有提升 → 保留最佳候选，进入下一轮
      current_video = best_candidate
      → 回到 Perceiver（观察新视频，生成新候选指令）
    else:
      # 无提升 → 回退到上一轮版本
      current_video = prev_video
      → 终止 或 Perceiver 尝试不同方向的指令

  终止条件：
    - 达到最大迭代次数（如 5 轮）
    - 连续 N 轮（如 2 轮）无评分提升

  一致性问题的自动注入：
    if consistency_score < threshold:
      → 在下一轮 Perceiver 的 prompt 中注入提示：
        "注意：当前各段色彩风格不一致，请优先考虑全局统一"
      → 引导 Agent 生成色彩统一指令
```

## 2.8 Evaluator 奖励模型：训练方法与数据来源

### 2.8.1 对标 PhotoAgent 的 Evaluator 架构

PhotoAgent 的 Evaluator = 4 个现成模型集成（CLIP + LAION Aesthetic + ImageReward）+ 1 个自训练 UGC Reward Model（基于 Qwen2.5-VL + GRPO，在 7,000 张 UGC 照片上训练）。

VlogAgent 采用相同思路：**现成模型覆盖大部分评估维度，仅对唯一缺失的"编辑美学"维度自训练奖励模型**。

```
VlogAgent Evaluator（5 维分解）

维度 ① 视觉质量      → 现成模型：DOVER + Q-Align           ← 无需训练
维度 ② 内容保真度     → 现成模型：CLIP 余弦相似度           ← 无需训练
维度 ③ 段间一致性     → 规则计算：Lab 色彩差异 + SSIM       ← 无需训练
维度 ④ 音频完整性     → 规则计算：语音区间检查 + 能量检测   ← 无需训练
维度 ⑤ 编辑美学评分   → 需要自训练奖励模型                  ← 唯一需要训练的
```

### 2.8.2 维度 ⑤ 的三阶段训练路径

#### Phase 0-1：MLLM-as-Judge 冷启动（零训练数据）

直接用 GPT-4o / Qwen-VL 作为评委打分，不训练任何模型：

```python
def mllm_judge(storyboard_before, storyboard_after, edit_actions):
    prompt = """
    你是专业视频编辑评审。对比编辑前后的视频关键帧，评估编辑质量。

    编辑前关键帧：[image: storyboard_before]
    编辑后关键帧：[image: storyboard_after]
    执行的编辑动作：{edit_actions}

    请从以下维度评分（1-5 分）：
    1. 视觉美感提升
    2. 编辑合理性
    3. 一致性
    4. 整体观感
    以 JSON 格式输出各维度评分和总分。
    """
    return gpt4o.call(prompt, images=[storyboard_before, storyboard_after])
```

同时在系统运行过程中**自动积累偏好对数据**（见 2.8.3 节）。

#### Phase 2：训练轻量奖励模型（~4,600 组偏好对后）

```
推荐方案：CLIP + MLP（训练最快，数据需求最小）

class VlogEditRewardModel(nn.Module):
    clip = CLIPModel("openai/clip-vit-large-patch14")
    mlp = nn.Sequential(
        Linear(768 * 2, 512), ReLU(),   # 编辑前后 CLIP 特征拼接
        Linear(512, 128), ReLU(),
        Linear(128, 1)                   # 输出编辑质量分
    )

    def forward(self, storyboard_before, storyboard_after):
        feat = concat(clip(before), clip(after))
        return mlp(feat)

训练损失：Pairwise Ranking Loss
    loss = -log(sigmoid(score_better - score_worse))

输入：编辑前后的 Storyboard 拼图对
输出：编辑质量分数
```

替换掉高频场景的 MLLM-as-Judge 调用，MLLM 降级为低频校准角色。

#### Phase 3：可选升级到 Qwen2.5-VL + GRPO

如果 CLIP+MLP 效果不够，升级到与 PhotoAgent 完全一致的方案：
- 基座：Qwen2.5-VL-7B
- 训练：GRPO（Group Relative Policy Optimization，学习组内相对排名）
- 50-100 组人工 A/B 偏好标注用于验证

### 2.8.3 奖励模型训练数据来源

**核心原则：不需要从零构建大规模标注数据集。** PhotoAgent 也只复用了已有的 LAION/RealQA 美学评分。VlogAgent 的偏好对数据在系统运行过程中**自然产出**。

#### 数据来源 1：MCTS 过程中的天然偏好对

```
MCTS 每轮比较多个候选动作序列：
  序列 A: 提亮→天空替换→统一色彩 → score=4.5
  序列 B: 稳定→提亮→移除路人     → score=4.3
  序列 C: 提亮→LUT→锐化          → score=4.2

→ (A > B), (A > C), (B > C) = 3 组偏好对
→ 每次 MCTS 搜索产生 ~20-50 组偏好对
```

#### 数据来源 2：Evaluator 的 accept/reject 记录

```
每轮闭环迭代中 Evaluator 判断 accept（分数提升）或 reject（回退）：
  第 1 轮编辑结果: score=3.8 > 原始 3.2 → accept
  第 2 轮编辑结果: score=4.1 > 3.8 → accept
  第 3 轮编辑结果: score=3.9 < 4.1 → reject

→ accept 版本 > 其前一版本 = 偏好对
→ reject 版本 < 其前一版本 = 偏好对
→ 每个视频的闭环过程产生 ~3-5 组偏好对
```

#### 数据来源 3：MLLM-as-Judge 打分数据

```
GPT-4o 对每个候选版本的 1-5 分评分：
  候选 A: 4.2 分
  候选 B: 3.8 分
  候选 C: 3.5 分

→ 高分 > 低分 = 偏好对
→ 每次评估产生 ~5-10 组偏好对
```

#### 数据来源 4：人工 A/B 标注（小规模校准）

```
展示两个编辑版本的 Storyboard 拼图
让标注者选"哪个编辑效果更好"
→ 50-100 组人工偏好对，用于验证模型 vs 人类一致性
```

#### 数据规模估算

| 来源 | 每个视频产出 | 100 个视频累计 |
|------|------------|--------------|
| MCTS 偏好对 | ~30 组 | ~3,000 组 |
| Accept/Reject | ~5 组 | ~500 组 |
| MLLM 打分 | ~10 组 | ~1,000 组 |
| 人工标注 | — | ~100 组（校准） |
| **合计** | | **~4,600 组偏好对** |

### 3.2 阶段 B 新增的核心问题：多素材拼接后的视觉一致性

这是阶段 B 相比阶段 A **最大的新增难题**。不同素材在不同时间、设备、环境下拍摄，即使各自美化后效果不错，拼在一起会出现亮度闪烁、色彩抖动、画质跳跃。

#### 3.2.1 问题分类

```
多素材一致性问题：

  类型 1 — 色彩空间差异（最常见）
    素材 A: iPhone 室内暖光, 色温 3200K
    素材 B: GoPro 室外日光, 色温 5600K
    拼接后：暖→冷的色彩突变

  类型 2 — 曝光/对比度差异
    素材 A: 曝光正常, 高对比
    素材 B: 略欠曝, 低对比
    拼接后：亮→暗的亮度跳变

  类型 3 — 分辨率/画质差异
    素材 A: 4K, 清晰锐利
    素材 B: 1080p, 有噪点
    拼接后：清晰→模糊的感知落差

  类型 4 — 帧率差异
    素材 A: 30fps
    素材 B: 60fps
    拼接后：流畅度不一致
```

#### 3.2.2 全局色彩统一策略（Color Harmonization）

```
Step 1: 锚点选择
  从所有已选素材中，选出"视觉质量综合评分最高"的片段
  选择标准：DOVER 评分 × 曝光正常度 × 色彩自然度
  → 作为全局色彩参考锚点

Step 2: 色彩特征提取（Lab 空间）
  对每个素材提取：
  - L 通道均值/方差（亮度分布）
  - a 通道均值/方差（红绿轴）
  - b 通道均值/方差（黄蓝轴）

Step 3: Reinhard Color Transfer
  对每个非锚点素材：
  target_L = (source_L - mean_source_L) × (std_anchor_L / std_source_L) + mean_anchor_L
  target_a = ... (同理)
  target_b = ... (同理)

  对齐强度可调（α ∈ [0,1]），默认 0.6

Step 4: 微调补偿
  Color Transfer 后可能导致某些素材失去原始特色
  → 由 Critic Agent 评估是否需要降低对齐强度
```

#### 3.2.3 段间过渡策略

```
策略 1 — 转场遮盖（推荐）
  使用 Crossfade / Dissolve 转场（0.5-1s）
  在过渡区间内两段画面自然融合
  副作用：总时长略微缩短

策略 2 — 拼接点色彩渐变
  在硬切点前后各 0.5-1s，对色彩参数做线性插值
  适合需要保留硬切节奏感的场景

策略 3 — 自适应策略
  由 Critic Agent 根据相邻段的视觉差异度选择：
  差异小（CLIP 相似度 > 0.85）→ 硬切即可
  差异中（0.7-0.85）→ 短 crossfade (0.3s)
  差异大（< 0.7）→ 长 dissolve (0.8s) 或 创意转场
```

### 3.3 阶段 B 的完整问题清单与解法汇总

| # | 问题类别 | 具体问题 | 解法 | 优先级 |
|---|---------|---------|------|--------|
| B1 | 搜索空间 | 多素材选择/排序的组合爆炸 | 元数据空间 MCTS + LLM 启发式剪枝 | 🔴 高 |
| B2 | 叙事评估 | "好的 Vlog 叙事"如何自动评分 | 分解式奖励 + MLLM-as-Judge | 🔴 高 |
| B3 | 跨素材色彩 | 不同素材拼接后色彩抖动 | 锚点 + Reinhard Color Transfer | 🔴 高 |

| B5 | 跨素材画质 | 4K 和 1080p 素材混合 | 统一到最低分辨率或超分辨率 | 🟡 中 |
| B6 | 跨素材帧率 | 30fps 和 60fps 混合 | 统一帧率转换（ffmpeg fps filter） | 🟡 中 |

| **色彩统一** | OpenCV (Lab Color Transfer) | Reinhard Color Transfer |

### AI Tools（AI 模型）

| 组件 | 选型 | 用途 |

| **视频物体移除** | ProPainter / E2FGVI | 移除路人/杂物/水印 |

### Agent 核心组件

| 组件 | 选型 | 用途 |
|------|------|------|
| **Perceiver (VLM)** | GPT-4o / Qwen-VL-Max | 观察视频 → 生成候选编辑指令 |

### Evaluator 组件（5 维分解评估）

| 评估维度 | 模型/方法 | 是否需要训练 |
|---------|----------|:---:|
| **视觉质量** | DOVER + Q-Align | 否（直接用） |
| **内容保真度** | CLIP 余弦相似度 | 否（直接用） |
| **段间一致性** | Lab 色彩差异 + SSIM | 否（规则计算） |
| **音频完整性** | 语音区间检查 + 能量检测 | 否（规则计算） |
| **编辑美学** | Phase 0-1: MLLM-as-Judge (GPT-4o) → Phase 2+: CLIP+MLP 自训练奖励模型 | 是（偏好对训练） |

## 六、风险与缓解

| 风险 | 概率 | 影响 | 缓解策略 |
|------|------|------|---------|
| MCTS 在动作序列空间收敛慢 | 中 | 延期 | LLM 启发式 Rollout + Top-K 执行缓冲；备选：贪心搜索 + 闭环迭代 |
| 段间一致性处理效果不佳 | 中 | 质量 | 三种策略（锚点/LUT/过渡平滑）A/B 测试；Perceiver 自主发现并修复 |
| 美学评估主观性导致实验方差大 | 中 | 论文 | 增加评估者数量 (20+)；报告 Fleiss' Kappa；多组 case study |
| MLLM API 成本过高 | 中 | 预算 | Phase 2 训练 CLIP+MLP 奖励模型替换高频 MLLM 调用；元数据空间规划减少调用量 |
| 奖励模型偏好对数据不足 | 低 | 质量 | MCTS 过程自动积累偏好对（100 个视频 ≈ 4,600 组）；必要时增加 MLLM 打分轮次 |
| 自训练奖励模型泛化性差 | 中 | 论文 | CLIP+MLP 不够则升级到 Qwen2.5-VL + GRPO；在外部数据集（YouTube UGC）上验证泛化 |
| 缺乏合适的测试素材 | 低 | 延期 | Pexels/Pixabay 免费素材 + YouTube CC BY + 自行拍摄（三条来源互补） |
| AI 工具生成质量不稳定 | 中 | 质量 | Evaluator 自动检测质量下降 → 回退；优先 Basic Tools，AI Tools 仅用于语义编辑 |
