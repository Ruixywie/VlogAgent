# VlogAgent 美学评估模型训练数据方案

## 一、目标

训练一个本地 CLIP+MLP 美学评估模型，替代当前 pipeline 中 MCTS 模拟和 Evaluator 美学评分的 VLM API 调用，将每次运行的 VLM 调用从 ~60-70 次降到 ~6 次（仅保留 Perceiver）。

## 二、模型架构

```
编辑前关键帧 → CLIP ViT-B-32 (冻结) → 特征向量 (512维)
编辑后关键帧 → CLIP ViT-B-32 (冻结) → 特征向量 (512维)
                                           ↓
                                     拼接 (1024维)
                                           ↓
                              MLP: 1024 → 256 → 64 → 1
                                           ↓
                                     编辑质量分 [0,1]

CLIP 冻结不训练，只训练 MLP（参数量 ~28 万，训练极快）
```

训练损失：Pairwise Ranking Loss
```
loss = -log(sigmoid(score_better - score_worse))
```

推理速度：单次评估 < 10ms（GPU）/ < 50ms（CPU），相比 VLM API 3-5 秒提速 100 倍。

## 三、数据来源规划

### 3.1 数据源 1：合成编辑对（核心数据，零 API 成本）

**原理**：从免费视频中提取帧，用 FFmpeg 对每帧施加不同强度的编辑，根据编辑强度自动标注质量。

**素材来源**：
- Pexels Videos: https://www.pexels.com/videos/ （免费商用）
- Pixabay Videos: https://pixabay.com/videos/ （免费商用）
- 目标：下载 100-200 段不同场景的短视频（旅行/城市/自然/美食/人物）

**生成流程**：

```
对每段视频：
  1. 每 2 秒提取 1 帧作为原始帧
  2. 对每帧施加多种编辑组合，生成编辑后帧
  3. 根据编辑参数自动标注质量等级
```

**编辑参数与标注规则**：

| 编辑类型 | 好的编辑（标签=正） | 差的编辑（标签=负） | 过度编辑（标签=强负） |
|----------|-------------------|-------------------|---------------------|
| 亮度 | brightness ∈ [-0.05, 0.08] | brightness ∈ [-0.15, 0.15] | brightness ∈ [-0.3, 0.3] |
| 对比度 | contrast ∈ [1.0, 1.15] | contrast ∈ [0.7, 1.3] | contrast ∈ [0.5, 2.0] |
| 饱和度 | saturation ∈ [1.0, 1.2] | saturation ∈ [0.7, 1.4] | saturation ∈ [0.5, 2.0] |
| 锐化 | amount ∈ [0.3, 0.8] | amount ∈ [1.0, 1.5] | amount ∈ [2.0, 3.0] |
| 降噪 | strength ∈ [2, 4] | strength ∈ [6, 8] | strength ∈ [9, 10] |
| 色温 | temp ∈ [5500, 7500] | temp ∈ [4000, 9000] | temp ∈ [2000, 10000] |

**质量标签映射**（自动标注，不需要人工）：

```
好的编辑（温和、合理的参数）  → quality = 0.7 - 0.9
轻微编辑（几乎无变化）       → quality = 0.5 - 0.6
差的编辑（参数偏激）         → quality = 0.3 - 0.4
过度编辑（严重失真）         → quality = 0.0 - 0.2
无编辑（原始帧 vs 原始帧）   → quality = 0.5（基准）
```

**组合编辑**（更贴近真实场景）：

```
组合 1（自然提亮）：brightness=0.05 + contrast=1.05 + saturation=1.1   → 0.75
组合 2（电影感）：  brightness=-0.02 + contrast=1.1 + saturation=0.9    → 0.80
组合 3（过度处理）：brightness=0.2 + contrast=1.5 + sharpen=2.0        → 0.15
组合 4（偏色）：    white_balance=3000 + saturation=1.5                → 0.10
```

**数据规模估算**：

```
100 段视频
× 每段提取 ~10 帧
× 每帧 ~15 种编辑变体（含单项 + 组合）
= 15,000 组 (原始帧, 编辑帧, 质量分) 三元组
→ 转为偏好对：C(15,2) 取子集 ≈ 50,000+ 组偏好对
```

**实现方式**：纯本地 FFmpeg + OpenCV + PIL 操作，不需要任何 API。

### 3.2 数据源 2：公开视频质量数据集（补充泛化能力）

用于让模型学习"什么是高质量视频帧"的通用审美能力。

#### KoNViD-1k

- **来源**: https://database.mmsp-kn.de/konvid-1k-database.html
- **规模**: 1,200 段视频
- **标注**: 每段有 MOS（Mean Opinion Score）主观质量分
- **用途**: 训练模型的基础视觉质量判断
- **使用方式**:
  - 从每段视频提取中间帧
  - MOS 分归一化到 [0, 1]
  - 作为"单帧质量评估"的训练数据
  - 高 MOS 帧 vs 低 MOS 帧 = 偏好对

#### LIVE-VQC

- **来源**: https://live.ece.utexas.edu/research/LIVEVQC/
- **规模**: 585 段手机拍摄视频
- **标注**: DMOS（差异主观质量分）
- **用途**: UGC（用户生成内容）场景的质量判断
- **使用方式**: 同 KoNViD-1k

#### YouTube-UGC Dataset

- **来源**: https://media.withyoutube.com/
- **规模**: 1,500 段视频
- **标注**: 质量分
- **用途**: 真实 YouTube 内容的质量基准
- **使用方式**: 同上

#### AVA（Aesthetic Visual Analysis）

- **来源**: https://github.com/mtobeiyf/ava_downloader
- **规模**: 255,530 张图片
- **标注**: 美学评分 1-10（众包标注）
- **用途**: 大规模美学感知预训练
- **使用方式**:
  - 选取评分分布的两端（高分 >7 和低分 <4）
  - 高分 vs 低分 = 偏好对
  - 用于预训练 MLP 的美学感知能力
  - 注意：AVA 是图片数据集，但 CLIP 特征对图片/视频帧通用

**公开数据集的总数据量**:

| 数据集 | 可提取帧数 | 偏好对估计 |
|--------|-----------|-----------|
| KoNViD-1k | ~1,200 帧 | ~5,000 对 |
| LIVE-VQC | ~585 帧 | ~2,000 对 |
| YouTube-UGC | ~1,500 帧 | ~6,000 对 |
| AVA（子集） | ~20,000 帧 | ~50,000 对 |
| **合计** | ~23,000 帧 | ~63,000 对 |

### 3.3 数据源 3：Pipeline 运行数据（精调 + 验证）

从实际 VlogAgent 运行中收集的真实评分数据，用于领域适配微调。

**收集内容**（每次运行自动保存）：

```python
{
    "video_id": "test_001",
    "iteration": 1,
    "plan_id": 1,
    "actions": ["color_adjust(brightness=0.05)", "auto_color_harmonize"],
    "original_frame_paths": ["seg0_mid.jpg", "seg1_mid.jpg", ...],
    "edited_frame_paths": ["seg0_mid_edited.jpg", "seg1_mid_edited.jpg", ...],
    "vlm_aesthetic_score": 0.72,
    "overall_score": 0.695,
    "accepted": true
}
```

**偏好对构建**：

```
同一迭代的 3 个方案：
  方案1: vlm_score=0.72  ← 最优
  方案2: vlm_score=0.58
  方案3: vlm_score=0.65

→ 偏好对：(方案1, 方案2), (方案1, 方案3), (方案3, 方案2)

跨迭代的 accept/reject：
  迭代1 accept (score=0.72) vs 原始 (score=0.50)
  → 偏好对：(迭代1结果, 原始)
```

**预期规模**：
- 每个视频产出 ~30 组偏好对
- 跑 20 个视频 = ~600 组偏好对
- 成本：约 20M token（约 20 次完整运行）

**用途**：在合成数据 + 公开数据集预训练后，用这 600 组真实数据做领域适配微调。

## 四、数据处理流程

### 4.1 统一数据格式

所有数据源统一为以下格式：

```python
@dataclass
class PreferencePair:
    """偏好对：better 的编辑质量优于 worse"""
    better_original_frame: str    # 原始帧路径
    better_edited_frame: str      # 更好的编辑帧路径
    worse_original_frame: str     # 原始帧路径（通常同上）
    worse_edited_frame: str       # 更差的编辑帧路径
    source: str                   # "synthetic" / "konvid" / "pipeline"
    margin: float                 # 质量差距（用于加权）
```

### 4.2 CLIP 特征预提取

训练前一次性提取所有帧的 CLIP 特征并缓存，避免训练时重复计算：

```
所有帧图片 → CLIP ViT-B-32 → 512维特征向量 → 保存为 .npy 文件

预提取时间估算：
  50,000 帧 × ~5ms/帧 = ~4 分钟（GPU）
```

### 4.3 数据划分

```
合成数据（15,000 组）     → 80% 训练 / 20% 验证
公开数据集（23,000 帧）   → 80% 训练 / 20% 验证
Pipeline 数据（600 组）   → 50% 微调 / 50% 测试

训练阶段：
  阶段 1：合成数据 + 公开数据集预训练（~60,000 偏好对）
  阶段 2：Pipeline 真实数据微调（~300 偏好对）
  阶段 3：Pipeline 测试集验证（~300 偏好对）
```

## 五、模型训练方案：两阶段递进

### 总体路线

```
Phase 2a：CLIP+MLP（快速验证，1-2 天完成）
  → 先替代 MCTS 模拟的 VLM 调用
  → 如果 Pairwise Accuracy > 85% 且端到端效果 OK → 同时替代 Evaluator 美学
  → 如果精度不够（漏判锯齿/伪影等局部问题）↓

Phase 2b：Qwen2.5-VL-7B + GRPO（完整方案，对齐 PhotoAgent）
  → 替代所有评估 VLM 调用
  → 可输出分数 + 理由（可解释性）
```

### 两方案对比

| | CLIP+MLP | Qwen2.5-VL-7B + GRPO |
|--|----------|----------------------|
| **能力** | 全局特征对比，判断整体好坏 | 理解画面细节，能发现局部伪影/锯齿 |
| **训练数据** | ~3,000 组偏好对 | ~7,000+ 组偏好对 |
| **训练资源** | 任意 GPU，几分钟 | 24GB+ 显存，训练几小时 |
| **推理速度** | <10ms/次 | ~200ms/次 |
| **推理显存** | ~500MB（CLIP） | ~16GB（7B 模型） |
| **输出** | 仅分数 | 分数 + 文字理由（可解释） |
| **局限** | 局部细节可能丢失 | 基本无，接近 VLM API 能力 |
| **适用场景** | MCTS 快速筛选 | 精确美学评估 |

### 5.1 Phase 2a：CLIP+MLP

#### 模型架构

```
编辑前关键帧 → CLIP ViT-B-32 (冻结) → 512维
编辑后关键帧 → CLIP ViT-B-32 (冻结) → 512维
                                         ↓
                                   拼接 (1024维)
                                         ↓
                          MLP: 1024 → 256 → 64 → 1
                                         ↓
                                   编辑质量分 [0,1]

可训练参数：~28 万（仅 MLP）
CLIP 冻结不训练
```

#### 训练配置

```
阶段 1：预训练（合成数据 + 公开数据集）
  数据：~60,000 组偏好对
  损失：Pairwise Ranking Loss = -log(sigmoid(s_better - s_worse))
  优化器：Adam, lr=1e-3, weight_decay=1e-4
  批大小：256
  训练轮数：20-30 epochs
  时间：~10 分钟（单 GPU）

阶段 2：微调（Pipeline 真实数据）
  数据：~300 组偏好对（来自 20 次 pipeline 运行）
  学习率：1e-4（降低 10 倍）
  训练轮数：10 epochs
  时间：~2 分钟
```

#### 验证指标

```
通过标准（可以部署替代 VLM）：
  - Pairwise Accuracy > 85%（偏好对预测准确率）
  - Kendall τ > 0.7（与 VLM 打分的排名一致性）

不通过标准（需要升级到 Phase 2b）：
  - Pairwise Accuracy < 80%
  - 端到端测试发现漏判明显质量问题（锯齿、色偏等）
```

#### 部署策略

```
先替代 MCTS 模拟（对精度要求较低，主要做粗筛）：
  MCTSPlanner._simulate() → CLIP+MLP 本地推理
  效果：MCTS VLM 调用 8-12 次 → 0 次

如果精度达标，再替代 Evaluator 美学评分：
  Evaluator.score_aesthetic() → CLIP+MLP 本地推理
  效果：Evaluator VLM 调用 ~10 次 → 0 次

推理方式：
  输入：编辑前后各 N 帧 → CLIP 特征 → MLP → 分数
  多帧聚合：对每段取中间帧，各段分数取平均
```

### 5.2 Phase 2b：Qwen2.5-VL-7B + GRPO

如果 CLIP+MLP 精度不够（尤其是漏判局部质量问题），升级到完整方案。

#### 模型架构

```
Qwen2.5-VL-7B（基座模型，支持视觉输入）
  ↓ GRPO 训练（Group Relative Policy Optimization）
  ↓
输入：编辑前关键帧 + 编辑后关键帧 + 评估 prompt
输出：JSON {"score": 0.0-1.0, "reason": "..."}

可训练参数：~7B（LoRA 微调时 ~10M 可训练参数）
```

#### 训练方法：GRPO（对齐 PhotoAgent）

```
GRPO 核心思想：
  对同一输入，模型生成一组回答（group），
  按照奖励信号对组内回答排序，
  让模型学习偏好高分回答。

具体流程：
  1. 输入：(编辑前帧, 编辑后帧, "请评估编辑效果")
  2. 模型生成 K=4 个回答（不同 temperature 采样）
  3. 用已有的偏好对数据计算每个回答的奖励
  4. GRPO 损失：让模型偏好高奖励回答
```

#### 训练配置

```
基座模型：Qwen2.5-VL-7B-Instruct
微调方式：LoRA（rank=16, alpha=32）
数据：~7,000+ 组偏好对
  - 合成编辑对：~5,000 组
  - 公开数据集：~1,500 组
  - Pipeline 真实数据：~500 组（跑到这个阶段应该积累更多了）
优化器：AdamW, lr=5e-5
批大小：4（梯度累积 8 步 = 有效批大小 32）
训练轮数：3-5 epochs
显存需求：24GB+（LoRA + 4-bit 量化可降到 16GB）
时间：~3-6 小时（单卡 A100/4090）
```

#### 训练数据格式

```json
{
  "messages": [
    {
      "role": "system",
      "content": "你是视频编辑质量评审专家。评估编辑前后的画面变化。"
    },
    {
      "role": "user",
      "content": [
        {"type": "text", "text": "原始画面:"},
        {"type": "image_url", "image_url": {"url": "原始帧"}},
        {"type": "text", "text": "编辑后画面:"},
        {"type": "image_url", "image_url": {"url": "编辑帧"}},
        {"type": "text", "text": "请评估编辑效果。返回 JSON: {\"score\": 0-1, \"reason\": \"...\"}"}
      ]
    }
  ],
  "preference_label": 0.72  // 来自 VLM 打分或偏好对排序
}
```

#### GRPO 训练依赖

```
pip install trl>=0.8.0        # Hugging Face TRL（含 GRPO 实现）
pip install peft>=0.10.0      # LoRA 微调
pip install bitsandbytes      # 4-bit 量化（降显存）
pip install transformers>=4.40
```

#### 验证指标

```
通过标准：
  - Pairwise Accuracy > 90%
  - Kendall τ > 0.8（与 VLM API 打分的排名一致性）
  - 端到端测试：模型版本输出 ≥ VLM 版本输出质量
```

#### 部署策略

```
完全替代所有评估 VLM 调用：
  MCTSPlanner._simulate() → Qwen-VL-7B 本地推理
  Evaluator.score_aesthetic() → Qwen-VL-7B 本地推理

效果预期：
  - VLM API 调用：~70 次 → ~6 次（仅 Perceiver 的观察+建议）
  - Token 消耗：~800k → ~150k（降 80%）
  - 运行时间：~30 分钟 → ~8 分钟（降 73%）
  - 推理速度：API 3-5s/次 → 本地 200ms/次

可选进一步优化：
  - vLLM 部署加速：推理速度可再提升 3-5 倍
  - 量化到 4-bit：显存降到 8GB，可在消费级 GPU 运行
```

Phase 2a（CLIP+MLP）阶段积累的所有数据，Phase 2b 可以直接复用：

## 八、验证方法

### Phase 2a 验证（CLIP+MLP）

1. **离线验证**：测试集 Pairwise Accuracy > 85%，Kendall τ > 0.7
